현재 상태 요약

CV32E40P는 OpenHW CV32E40P RTL을 ASIC 구현 flow에 맞게 wrapper, technology RTL, filelist, constraint로 정리한 뒤 Synopsys Front-End flow를 끝까지 통과시킨 학습 프로젝트다. 1차 기준 run은 tt_mvt_10ns_scan1이고, 이후 clean baseline으로 8.5 ns Front-End closure까지 확인했다.

Front-End 관점의 핵심 결론은 다음과 같다. 8.5 ns clean mixed-VT flow는 DC topo synthesis, Formality R2N, DFT insertion, Formality N2N, TetraMAX stuck-at ATPG, PrimeTime post-DFT SDF STA까지 완료됐다. known notes는 DFT TEST-505, TetraMAX Z3, backend-defer 성격의 early electrical DRC다.

1. 프로젝트 목표와 학습 범위

CV32E40P를 선택한 이유는 단순한 toy RTL보다 실제 CPU RTL이 갖는 integration 문제를 학습하기 좋기 때문이다. CPU RTL에는 package/filelist ordering, clock gating, debug/CSR 구조, scan insertion, large sequential state, timing constraints, ATPG protocol 같은 실제 구현 flow의 핵심 문제가 포함된다.

프로젝트의 첫 목표는 “RTL을 합성했다”가 아니라 “Front-End closure evidence를 만들었다”이다. 따라서 각 stage마다 입력, script, output, report, pass/fail 기준을 남겼고, 결과 판단은 chat이나 terminal 감상이 아니라 report artifact에 연결했다.

3. SDC와 Timing 목표

TEST-505 해석

TEST-505는 clock-gate test control이 constant 성격으로 보이는 구조에서 나오는 DFT note다. 이 프로젝트에서는 scan cell count, chain tracing, ATPG coverage가 정상이고 Formality N2N도 통과했기 때문에 first-pass educational flow의 blocker로 보지 않았다. 단, production DFT signoff에서는 clock-gate controllability와 test mode intent를 더 엄밀히 검토해야 한다.

이 단계에서 다음 단계로 넘어간 이유: post-DFT netlist, SPF, SDF가 생성됐고 scan chain 1개와 chain length 2130이 report에 기록됐다. fatal DFT DRC가 없고 known TEST-505 note만 남았으므로, pre/post-DFT functional equivalence 확인으로 넘어갔다.

7. Formality N2N

N2N은 pre-DFT netlist와 post-DFT netlist의 functional equivalence를 확인한다. scan insertion은 테스트 구조를 추가하지만, functional mode에서는 원래 동작과 같아야 한다. 따라서 scan/test inputs를 constant로 고정하고 scan_out은 architectural output이 아니므로 don't-verify로 처리했다.

8. TetraMAX Stuck-At ATPG

ATPG는 scan 구조가 실제로 test pattern generation에 사용 가능한지 검증한다. 이 프로젝트는 stuck-at model을 first-pass target으로 삼았다. TetraMAX는 SAED32 test model, post-DFT netlist, SPF를 읽고 chain0을 trace했다.

Z3 warning 해석

Z3는 wire contention 가능성 rule이다. 이 프로젝트에서는 6개 Z3 warning을 warning으로 낮추어 first-pass stuck-at ATPG를 진행했다. ATPG coverage와 scan tracing이 정상이라고 해서 Z3가 production signoff에서 무시된다는 뜻은 아니다. 내부 학습 flow에서는 “ATPG를 진행하는 데 blocking fatal은 아니다”로 분류했다.

이 단계에서 다음 단계로 넘어간 이유: SPF chain tracing이 성공했고 stuck-at test/fault coverage가 98% 이상이었다. Z3는 known warning으로 기록했지만 ATPG fatal이 아니었기 때문에 post-DFT timing validation으로 넘어갔다.

9. PrimeTime Post-DFT SDF STA

PrimeTime SDF STA는 DC/DFT Compiler가 출력한 SDF delay를 gate netlist에 annotate해 setup/hold를 확인한다. Backend routing 이전에는 real SPEF가 없으므로 이 STA는 post-synthesis/post-DFT timing handoff 품질을 보는 단계다.

표 11. Post-DFT SDF STA 결과
StageClockSetupHold판단
post-DFT topo/SDF10 nsWNS +1.48 nshold slack +0.03 nsFE baseline clean
post-DFT topo/SDF8.5 nsWNS +0.44 nshold slack +0.04 nsFE closure clean

SDF annotation errors가 0이라는 점도 중요하다. SDF instance/path mismatch가 있으면 timing report가 clean이어도 신뢰도가 떨어진다. 이 프로젝트는 SDF read 0 errors로 기록했다.

이 단계에서 다음 단계로 넘어간 이유: SDF annotation error가 0이고 setup/hold가 모두 clean이었다. 이로써 Front-End closure artifact가 backend handoff package로 넘어갈 수 있었다.

10. Backend Handoff Package

Front-End 결과가 backend로 넘어가기 위해서는 post-DFT logical netlist와 timing constraints, scan DEF/SPF/SDF가 정리되어야 한다. 이 프로젝트는 ICC2가 post-DFT netlist를 physical design으로 열 수 있는지 확인했고, scan DEF도 별도 script로 생성했다.

  • R2N/N2N PASS는 다음 stage debug 범위를 크게 줄인다.
  • DFT는 “scan inserted”만 보는 것이 아니라 SPF chain length와 TetraMAX tracing까지 확인해야 한다.
  • ATPG coverage는 test coverage, fault coverage, pattern count, DRC warning을 함께 읽어야 한다.
  • Post-DFT SDF STA가 clean이어도 backend route SPEF STA는 별도 문제다.
  • max_cap/max_transition은 Front-End note로 남기고 backend real parasitic에서 다시 보는 것이 맞다.
